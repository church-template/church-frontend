# Issue-Branch — 이슈 등록 + 작업 브랜치 생성

`/issue`로 이슈 문서를 만들고, GitHub에 등록해 **실제 이슈 번호**를 받은 뒤,
저장소 브랜치 관례대로 작업 브랜치를 만들어 전환하는 통합 커맨드.

## 실행 절차

### 1단계: 이슈 작성 (/issue 규칙 그대로)

`.claude/commands/issue.md`(Issue Mode) 규칙을 그대로 따른다:

- 타입 자동 판단(버그/기능추가·개선·요청/디자인/시험요청), 애매하면 짧게 질문
- 제목: `이모지 [타입][카테고리] 설명` — 본문은 해당 템플릿, 굵게 강조 금지, 담당자 기본값 유지
- `.issues/[YYYYMMDD]_[타입]_[간단한설명].md`로 저장

### 2단계: GitHub 등록 → 이슈 번호 확보

```bash
gh issue create --title "<제목>" --body "<본문(파일에서 제목 줄 제외)>"
```

출력 URL 끝의 번호가 **이슈 번호**다. 등록 실패(네트워크·권한) 시 파일만 남기고
사용자에게 알린 뒤 중단한다 — 번호 없이 브랜치를 만들지 않는다.

### 3단계: 작업 브랜치 생성·전환

브랜치 관례: `YYYYMMDD_#번호_한글_설명_언더스코어` (예: `20260717_#95_공개_페이지_이동_로딩_스켈레톤_추가`)

- 날짜 = 오늘, 번호 = 2단계에서 받은 번호, 설명 = 이슈 제목의 핵심을 언더스코어로
- 시작점: 현재 브랜치가 `main`이 아니면 사용자에게 확인 후 진행

```bash
git checkout -b "<브랜치명>"
```

### 4단계: 보고 후 그 브랜치에서 작업 계속

- 이슈 URL·번호 · `.issues/` 파일 경로 · 브랜치명을 표로 출력
- 이후 모든 커밋 메시지는 `<type> : <설명> #번호` 형식으로 이 번호를 붙인다

## 주의

- push 금지 · Co-Authored-By 금지 — 커밋은 사용자가 요청할 때만
- 이슈 본문에 코드·구현 방법 쓰지 않음(Issue Mode 규칙)
- `gh` 미인증이면 `gh auth status` 안내 후 중단
