# 문의(Inquiry) 프론트 연동 (#90)

### 📌 작업 개요

백엔드에 새로 추가된 문의 API 4종을 프론트와 연결. 방문자가 홈페이지에서 문의를 남기는 공개 접수 폼과, 담당자가 접수된 문의를 읽고 처리 여부를 기록하는 관리 화면까지 도메인 전 흐름을 구현.

두 가지 사실이 설계를 규정했다.

1. **공개 등록은 비인증**이다. 기존 쓰기 경로(`apiMutate`)는 전부 `authFetch`(401 refresh·토큰 큐잉) 위에 있어 그대로 쓸 수 없다 — 방문자는 토큰이 없다.
2. **목록 응답에 문의 내용(`content`)이 없다.** 관리자는 내용을 읽으려면 반드시 상세를 열어야 하므로, "목록에서 바로 완료 체크"는 내용을 안 읽고 체크하는 오조작을 부른다. 따라서 읽기와 처리를 같은 자리(상세 다이얼로그)에 묶었다.

- 설계 스펙: `docs/superpowers/specs/2026-07-14-inquiry-frontend-design.md`
- 구현 계획: `docs/superpowers/plans/2026-07-14-inquiry-frontend.md`

### 🎯 구현 목표

- 공개: 비로그인 방문자의 문의 접수(`/about/location`) — 개인정보 동의 필수, 접수번호 확인
- 관리자: `INQUIRY_MANAGE` 보유자의 문의 목록·상세·완료 처리·삭제(`/mypage/manage/inquiries`)
- 기존 패턴(태그·회원 관리 화면) 재사용 — 새 인프라·새 라이브러리 0

### ✅ 구현 내용

#### 1. API 연동 계층

| 파일 | 역할 |
|---|---|
| `src/lib/api/inquiries.ts` | 공개 등록(`POST /api/inquiries`). `authFetch`를 타지 않는 얇은 `fetch` + `parseJson` — 비회원 제출이라 토큰 refresh 체인에 들어가면 안 된다 |
| `src/lib/api/inquiries.admin.ts` | 어드민 목록·상세·완료 토글·삭제. 읽기는 `authFetch`+`parseJson`, 쓰기는 `apiMutate`(`members.admin.ts` 동형). 클라이언트 전용 |

- 목록 `completed` 미지정 = 전체(쿼리 파라미터 자체를 생략), `false`=미처리, `true`=완료
- 정렬은 백엔드 기본(최신순)에 위임 — `sort`를 보내지 않는다
- 낙관락(`version`) 없는 도메인

#### 2. 공개 문의 접수 폼

`src/components/about/InquirySection.tsx` + `inquirySchema.ts`

- 이름(≤50)·연락처·이메일(선택 ≤100)·문의 내용(10~2000)·개인정보 동의(필수) — zod 스키마가 백엔드 제약과 1:1
- 연락처 자동 하이픈(`formatPhone`), 개인정보처리방침 전문 보기(`TermsDialog`) — 가입 폼 조합 재사용
- **필수/선택은 라벨 텍스트로 표기**(`이름 (필수)` / `이메일 (선택)`) — 별표는 고령 사용자가 놓치기 쉽다. `required` 속성으로 `aria-required`도 부여(폼이 `noValidate`라 브라우저 팝업은 뜨지 않음)
- **제출 성공 시 폼을 접수 완료 패널로 교체**(접수번호 표시) — 토스트만으로는 고령 사용자가 접수 사실을 놓친다. `role="status"` + 제목 포커스 이동으로 보조기술에도 전달
- 실패는 `errorCode` 분기(`handleApiError`) — 필드 검증은 인라인, 그 외는 Toast

#### 3. 관리 화면

`src/app/(site)/mypage/manage/inquiries/page.tsx` + `src/components/admin/inquiries/*`

- **`InquiryManager`**: Tabs 필터(전체·미처리·완료 → URL `?completed=`) + `DataTable`(이름·연락처·접수일·상태 Badge) + URL 구동 `Pagination`(10건). 필터 변경 시 page 리셋. 이름 셀을 버튼으로 둬 키보드로도 상세를 연다
- **`InquiryDetailDialog`**: 상세 시드(`useQuery` 파생) + 연락처(`tel:`)·이메일(`mailto:`) + 본문 + 완료 처리/완료 취소 + 삭제(`DeleteConfirmDialog`). 본문은 방문자가 쓴 평문이라 마크다운으로 변환하지 않고 `whitespace-pre-wrap`으로 그대로 렌더
- 페이지 게이트: `RequirePermission permission="INQUIRY_MANAGE"`(서버가 2단 방어)

#### 4. 곁가지

- `constants/permissions.ts`: `INQUIRY_MANAGE` → "문의 관리" 라벨
- `lib/admin/manageDomains.ts`: 관리 허브에 **새 카테고리 `inbox`("문의")** 신설 + 카드 등록. 기존 4개 카테고리(콘텐츠·미디어·조직·회원권한)는 전부 "교회가 내보내는 것"이라 "들어오는 것"인 문의를 끼워 넣으면 의미가 어긋난다. 관리자가 가장 자주 확인할 항목이라 허브 맨 위
- `lib/auth/handleApiError.ts`: 429 `RATE_LIMIT_EXCEEDED` 분기 추가(IP당 1시간 5건 제한). 전역 공유 핸들러라 문구는 중립("요청이 너무 많습니다…")

#### 5. 연락처 페이지 UI 개선

- **연락처 항목 복사**(`CopyableField`): 주소·전화번호·이메일 각 행에 복사 버튼(48px 터치 타깃) 상시 노출 + 길게 누르기(500ms) 복사. 길게 누르기는 눈에 보이지 않는 제스처라 그것만으로는 알 수 없어 버튼과 안내 문구를 함께 둔다. 복사 후 2초간 체크 아이콘 + Toast, 클립보드 실패 시 "직접 선택해 복사" 안내(침묵 실패 방지). 길게 눌러 복사한 직후의 click(전화 앱 열림)은 1회 억제
- **레이아웃**: 연락처(흰 섹션) 다음 회색 밴드에 **좌 `찾아오는 방법` : 우 문의 카드 = 5:7** 2컬럼(모바일 세로 스택). 문의 폼의 짧은 필드(이름·연락처)는 2열 배치 — 한 줄에 하나씩 쌓으면 카드가 옆 컬럼보다 과하게 길어진다
- **헤더**: 교회명 옆에 로고 추가(장식이라 `alt=""` — 스크린리더가 교회명을 두 번 읽지 않게). 헤더 높이 80→96px, 메뉴 21→24px, 로고 40px, 교회명 30px로 확대(토큰 값만 변경 → 메가메뉴·모바일 시트·히어로가 자동 반영)

### ⚠️ 검토 과정에서 잡은 결함

구현 후 리뷰에서 실제 버그 3건이 확인되어 수정했다.

| 결함 | 증상 | 조치 |
|---|---|---|
| `ApiError` 인자 위치 오류 | 네트워크 단절 시 안내 문구가 `detail`에 실렸는데 `handleApiError`의 기본 분기는 `title`만 읽어, 실제로는 "오류가 발생했습니다."만 표시 | 문구를 `title` 자리로 이동 + 해당 경로 회귀 테스트 추가 |
| 삭제 후 삭제된 리소스 재조회 | 삭제 성공 시 상세 캐시를 `invalidateQueries` → 아직 다이얼로그가 열린 시점이라 삭제된 문의를 즉시 재조회(404) → "삭제했습니다" 직후 오류 Toast | 삭제 경로는 상세 캐시를 `removeQueries`로 제거(재조회 없음). 완료 토글 경로는 그대로 |
| 방문자 개인정보 캐시 잔존 | 상세 다이얼로그에 `gcTime` 미설정 → 창을 닫아도 이름·전화·이메일·본문이 5분간 캐시에 잔존 | `staleTime: 0, gcTime: 0` |

접근성 2건(필터 Tabs의 `aria-controls`가 존재하지 않는 패널을 가리키던 문제, 접수 완료 패널이 라이브 리전이 아니라 스크린리더가 읽지 못하던 문제)도 함께 수정했다.
