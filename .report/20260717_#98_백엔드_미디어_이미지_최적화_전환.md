# 백엔드 미디어 이미지 최적화·CDN 캐싱 적용

### 📌 작업 개요

백엔드 미디어 이미지(`/api/media/{id}`)가 일반 `<img>` 태그로 원본 그대로 전송되던 것을 next/image 최적화 파이프라인으로 전환. 기기 화면 폭에 맞는 크기·최신 포맷(WebP/AVIF) 변환본이 CDN에 캐싱되어 내려가도록 개선

**보고서 파일**: `.report/20260717_#98_백엔드_미디어_이미지_최적화_전환.md`

### 🎯 구현 목표

- 갤러리·어드민 화면의 백엔드 사진 전송량 절감 (원본 수 MB → 표시 크기에 맞는 수십 KB)
- 최적화 변환본의 CDN 캐싱으로 재방문·재조회 시 로딩 속도 개선
- 화질·레이아웃 변화 없이 적용 (기존 hover 줌·라운드·비율 유지)

### ✅ 구현 내용

#### 이미지 최적화 설정 추가
- **파일**: `next.config.ts`
- **변경 내용**: `images.remotePatterns`에 백엔드 미디어 경로(`/api/media/**`)만 허용하도록 설정. 오리진은 `NEXT_PUBLIC_API_BASE` 환경변수에서 파생. `minimumCacheTTL`을 31일로 설정
- **이유**: 오리진을 하드코딩하지 않아 개발(localhost)·운영(교회별 도메인) 모두 자동 대응 — 교회 템플릿 재사용성 유지. 미디어는 id 단위로 불변(수정 API 없음, 교체 = 새 업로드·새 id)이라 긴 캐시가 안전

#### 갤러리 앨범 카드 썸네일 전환
- **파일**: `src/components/gallery/AlbumCard.tsx`
- **변경 내용**: `<img>` → next/image `fill` 모드 + `sizes` 지정(데스크톱 33vw / 태블릿 50vw / 모바일 100vw), 썸네일 컨테이너에 `relative` 추가
- **이유**: 원본 크기를 알 수 없는 원격 이미지라 `fill` + 컨테이너 비율(16:9) 방식 사용. `sizes`는 앨범 목록 그리드(3/2/1열)와 일치시켜 필요한 크기만 다운로드

#### 앨범 상세 사진 그리드 전환
- **파일**: `src/components/gallery/PhotoGrid.tsx`
- **변경 내용**: `<img>` → next/image `fill` + `sizes`(25vw/33vw/50vw), 정사각 래퍼에 `relative` 추가
- **이유**: 사진 그리드(4/3/2열)가 갤러리에서 이미지가 가장 많은 화면 — 절감 효과가 가장 큰 지점

#### 어드민 미디어 선택 다이얼로그 전환
- **파일**: `src/components/admin/MediaPicker.tsx`
- **변경 내용**: `<img>` → `relative` 래퍼 + next/image `fill`, `sizes="200px"`(다이얼로그 셀 폭 근사)
- **이유**: 다이얼로그 내 고정 폭 그리드라 뷰포트 비율 대신 고정 크기 지정이 더 정확

#### 어드민 미디어 라이브러리 썸네일 전환
- **파일**: `src/components/admin/media/MediaLibrary.tsx`
- **변경 내용**: `<img>` → next/image `width={40} height={40}` 고정 크기
- **이유**: 40px 표시 자리에 원본을 통째로 받던 최악의 낭비 지점 — 고정 크기 지정으로 40px 변환본만 수신

#### 테스트 갱신
- **파일**: `src/components/gallery/AlbumCard.test.tsx`
- **변경 내용**: 썸네일 src가 최적화 프록시(`/_next/image?url=…`)를 경유하는지 단언하도록 수정
- **이유**: 일반 `<img>` 회귀를 테스트가 잡아내도록 보장

### 🔧 주요 변경사항 상세

#### 전환하지 않은 곳 (의도적 제외)

- `PhotoLightbox`(사진 확대 모달): 원본 감상 용도 + 이미지 크기에 따라 레이아웃이 결정되는 구조라 유지
- `SermonVideo`(유튜브 썸네일): 유튜브 자체 CDN이 이미 최적화 제공
- 정적 `public/` 에셋 컴포넌트(부서 카드·연혁·교회사진 등): 백엔드 미디어가 아니므로 이번 범위 밖
- 설교 목록: 현재 썸네일 데이터를 사용하지 않아 대상 없음 (SermonCard의 이미지 코드는 휴면 상태)

**특이사항**:
- `fill` 모드는 next/image 규약상 부모 요소에 `position: relative`가 필요 — 각 래퍼에 `relative` 클래스 추가
- `NEXT_PUBLIC_API_BASE` 미설정 환경(CI 빌드)에서는 이미지 src가 상대경로가 되어 원격 패턴이 불필요 — 설정에서 빈 배열로 처리
- 캐시 무효화 메커니즘이 없으므로(프레임워크 제약) 31일 TTL은 "미디어 id 불변" 전제에 의존. 백엔드에 미디어 수정 API가 생기면 TTL 재검토 필요
