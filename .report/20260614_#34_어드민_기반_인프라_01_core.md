# 어드민 기반 인프라 (01-core)

### 📌 작업 개요

어드민 트랙(2차 배치)의 첫 작업. 백엔드 `/api/admin/**` 쓰기 API는 열려 있으나 프론트 연동이 0%인 상태에서, 후속 도메인(02~07)이 공유할 **공통 토대**를 구축. 도메인 CRUD는 없고, 권한 게이트·쓰기 헬퍼·오류 매핑·삭제 다이얼로그·관리 허브만 만든다.

진입 모델은 별도 `/admin` 백오피스가 아니라 ① 공개 페이지 인라인 액션 ② 마이페이지 관리 허브 ③ 운영 도메인 전용 화면 세 갈래로 분산한다.

### 🎯 구현 목표

- 권한 보유 시에만 액션을 노출하는 RBAC 게이트
- 어드민 JSON 쓰기 공통 헬퍼와 useMutation 오류/무효화 규약
- 어드민 도메인 errorCode 처리 확장
- 파괴적 확인 다이얼로그
- 마이페이지 관리 허브 + `/mypage/manage` 라우트 스캐폴드

기존 자산(`authFetch`·`parseJson`·`ApiError`·`handleApiError`·`useMe`·`hasPermission`·Button `destructive`·Dialog·PasswordInput·MarkdownContent)을 **재구현 없이 확장**하는 것을 원칙으로 함.

### ✅ 구현 내용

#### RBAC 게이트
- **파일**: `src/components/admin/RequirePermission.tsx`, `src/lib/auth/useMe.ts`
- **변경 내용**: `<RequirePermission permission|perms mode fallback>` 컴포넌트 신규. `useMe()`의 라이브 `permissions`를 단일 기준으로 판정하고, 로딩 중·미보유 시 children 비렌더. 분기 로직용 `useHasAnyPermission` 추가, `useHasPermission`은 기존 `usePermission` 별칭으로 노출
- **이유**: 토큰 스냅샷이 아니라 실시간 권한으로 게이팅(가이드 1.5·2.1). 게이트는 UX 최적화일 뿐 보안 경계가 아니며, 서버가 2단 방어

#### 쓰기 헬퍼 + 뮤테이션 규약
- **파일**: `src/lib/admin/apiMutate.ts`, `src/lib/admin/mutationHandlers.ts`, `src/lib/admin/queryKeys.ts`
- **변경 내용**: `apiMutate<T>(path, {method, body})` — `authFetch` + `parseJson` 조합 JSON 쓰기 헬퍼. `adminOnError(handlers)` — ApiError는 `handleApiError` 위임, 그 외는 네트워크 오류 토스트. `isOptimisticLockConflict(e)` — 낙관락 충돌 판별. `adminKeys.list/detail` — `"admin"` 네임스페이스 쿼리키 규약
- **이유**: 02~07 도메인 쓰기가 동일한 패턴(클라이언트 + TanStack Query useMutation + authFetch)을 재사용하도록 통일

#### errorCode 처리 확장
- **파일**: `src/lib/auth/handleApiError.ts`
- **변경 내용**: 어드민 도메인 코드 3종(`ROLE_IN_USE`·`DEPARTMENT_HAS_CHILDREN`·`FILE_SIZE_EXCEEDED`)에 사용자 친화 고정 메시지 case 추가
- **이유**: 역할 삭제 불가·하위 부서 존재·파일 한도 초과를 명확히 안내. `FILE_STORAGE_ERROR`·`INTERNAL_ERROR`는 기존 default(서버 title 폴백)로 충분해 case 미추가

#### 삭제 확인 다이얼로그
- **파일**: `src/components/admin/DeleteConfirmDialog.tsx`
- **변경 내용**: Dialog + Button `destructive` + PasswordInput 조합. controlled(`open`/`onOpenChange`), `requirePassword` 시 비번 미입력이면 확정 비활성, 닫을 때 비번 초기화
- **이유**: 파괴적 작업의 친화 장치는 색이 아니라 경고문 + 비밀번호 재인증(DESIGN button-destructive)

#### 마이페이지 관리 허브 + 라우트 스캐폴드

- **변경 내용**: 도메인↔권한↔경로 매핑 상수(`MANAGE_DOMAINS` 11개) + 권한 보유 도메인만 카드로 노출하는 `ManageHub`(보유 0이면 섹션 비노출). `/mypage`에 조립. `/mypage/manage` 레이아웃은 로그인 가드만, 인덱스는 `/mypage`로 리다이렉트
- **이유**: 운영 도메인 화면(부서·미디어·태그·직분·회원·역할)이 들어올 부모 라우트를 미리 깔고, 권한별 진입을 한 곳에 모음

### 🔧 주요 변경사항 상세

#### apiMutate의 204 처리
`parseJson`은 2xx면 `res.json()`을 직접 호출하는데, DELETE의 204 No Content는 본문이 없어 빈 본문 파싱 에러가 난다. `apiMutate`가 `res.status === 204`를 먼저 가로채 `undefined`를 반환하도록 처리.

**특이사항**:
- FormData 업로드는 Content-Type 수동 설정 금지라 이 헬퍼를 쓰지 않음(미디어 업로더는 후속 도메인에서 별도 구현)

#### ManageHub의 등장 래퍼 흡수
관리 권한이 없는 일반 회원(대다수)에게 허브가 비노출될 때, 등장 애니메이션 래퍼(`Reveal`)가 빈 채로 남아 잉여 여백이 생기는 문제를 방지. `Reveal`을 `ManageHub` 내부로 흡수해 권한이 없으면 자리 자체가 사라지도록 구성.

#### 기존 테스트와의 정합
`handleApiError` 테스트가 `ROLE_IN_USE`·`INTERNAL_ERROR`를 default 분기 예시로 쓰고 있어, 신규 case 추가 시 `ROLE_IN_USE` 테스트는 case 동작으로 갱신하고 default 예시는 미정의 코드로 교체.
