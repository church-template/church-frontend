### 📌 작업 개요
교회 홈페이지를 홈 화면에 설치 가능한 PWA로 만드는 작업. Next.js 파일 컨벤션(`manifest.ts`, `apple-icon.png`)만으로 신규 의존성 없이 구현했고, 같은 이슈에 iOS Safari 하이드레이션 불일치 수정을 편승시켰다. 서비스워커·오프라인·푸시·설치 안내 UI는 범위 밖.

### 🎯 구현 목표
- 홈 화면 추가(Add to Home Screen) 시 앱 이름·아이콘·standalone 실행 모드가 적용되도록 manifest 제공
- Android maskable 아이콘·iOS apple-touch-icon 대응
- 앱 콘텐츠(이름·설명)는 하드코딩 없이 `church.ts` 상수에서 주입

### ✅ 구현 내용

#### manifest.ts — Next 파일 컨벤션 기반 PWA manifest
- **파일**: `src/app/manifest.ts`
- **변경 내용**: `MetadataRoute.Manifest`를 반환하는 기본 export 함수 추가. `name`·`short_name`·`description`은 `CHURCH_NAME`·`CHURCH_DESCRIPTION`(`@/constants/church`) 주입, `display: "standalone"`, `start_url: "/"`, `lang: "ko"`, `background_color`/`theme_color`는 `#ffffff`(DESIGN.md canvas 값)로 고정. `icons`는 192×192·512×512(any) + 512×512(maskable) 3개 배열
- **이유**: Next가 이 파일 하나로 `/manifest.webmanifest` 라우트와 `<link rel="manifest">` 주입을 자동 처리 — 별도 서비스워커·라이브러리 불필요

#### manifest.test.ts — TDD 테스트
- **파일**: `src/app/manifest.test.ts`
- **변경 내용**: vitest 테스트 3건 — (1) 이름·설명이 church.ts 상수와 일치하는지, (2) standalone·start_url·lang·배경/테마 색 값 검증, (3) 아이콘 배열이 any 2개 + maskable 1개 정확히 3개인지 `toEqual`로 검증
- **이유**: 프로젝트 관례(RED→GREEN)를 따라 구현 전 실패 테스트 작성

#### 아이콘 에셋 4종 생성
- **파일**: `public/icon-192.png`(192², 36.7KB) · `public/icon-512.png`(512², 171.9KB) · `public/icon-maskable-512.png`(512², 101.7KB) · `src/app/apple-icon.png`(180², 32.2KB)
- **변경 내용**: 기존 `public/onlyLogo.png`(512² 투명 PNG)를 소스로 macOS `sips`만 사용해 생성. icon-192/512는 단순 리사이즈(투명 배경 유지). maskable-512는 로고를 340px로 축소 후 512로 흰 배경 패딩. apple-icon은 180으로 리사이즈. maskable·apple 두 아이콘은 `sips`에 알파 평탄화 옵션이 없어 PNG→JPEG→PNG 왕복으로 투명 배경을 흰색으로 평탄화
- **이유**: Android는 maskable 아이콘을 원형/스쿼클로 마스킹하므로 로고 주변 안전영역 패딩이 없으면 잘림. iOS는 투명 배경을 검정으로 채우므로 apple-icon도 평탄화 필수

### 🔧 주요 변경사항 상세

#### manifest.ts
색상값(`#ffffff`)은 manifest가 CSS 변수를 읽을 수 없어 이 파일에서만 예외적으로 하드코딩했고, 출처(DESIGN.md canvas)를 주석으로 명시했다. 아이콘 경로는 문자열 리터럴로 직접 참조(Task 2에서 생성한 4개 에셋 중 apple-icon 제외 3개만 manifest 소비).

**특이사항**:
- 신규 npm 의존성 0개 — Next 16.2.9 내장 파일 컨벤션과 macOS `sips`(1회성 생성, 스크립트는 저장소에 남기지 않음)만 사용
- 아이콘 생성용 셸 스크립트는 계획 문서(`docs/superpowers/plans/2026-07-19-pwa-installable.md`)에만 기록, 커밋 대상 아님

#### iOS Safari 하이드레이션 불일치 수정 (같은 이슈 편승)
iOS Safari 시뮬레이터로 홈 화면 진입 확인 중 Recoverable Hydration Error 발견. 원인은 Safari의 전화번호 자동 감지 기능이 하이드레이션이 끝나기 전에 푸터의 평문 전화번호 텍스트를 `<a href="tel:...">`로 감싸며 서버 렌더 DOM과 클라이언트 DOM을 불일치시키는 것.

- **파일**: `src/app/layout.tsx`
- **변경 내용**: `metadata` 객체에 `formatDetection: { telephone: false }` 추가해 Safari의 자동 전화번호 감지·DOM 변조를 차단
- **이유**: 감지를 끄는 대신 통화 기능은 프론트가 명시적으로 제공해야 함

- **파일**: `src/components/shell/SiteFooter.tsx`
- **변경 내용**: 평문으로 렌더되던 `CHURCH_PHONE`을 `<a href={\`tel:${CHURCH_PHONE}\`}>` 링크로 교체
- **이유**: 자동 감지를 껐지만 탭 통화 기능 자체는 유지해야 하므로 프론트가 직접 tel: 링크 제공

- **파일**: `src/components/shell/SiteFooter.test.tsx`
- **변경 내용**: 전화번호가 `tel:` href를 가진 링크로 렌더되는지 검증하는 테스트 1건 추가(role="link" + accessible name으로 조회, href 값 단언)

**특이사항**:
- 증상 재현 환경이 iOS Safari 특유의 자동 감지 동작이라 일반 브라우저에서는 재현되지 않음 — 근본 원인을 `formatDetection` 전역 설정으로 해결해 향후 다른 컴포넌트에서 평문 전화번호를 노출해도 동일 문제가 재발하지 않도록 처리

### 📦 의존성 변경
