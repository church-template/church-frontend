# 공지 목록·검색·상세 (#12)

### 📌 작업 개요

교회 공지(notice) 공개 페이지 구현. **목록(검색·태그 필터·페이지네이션)** 과 **상세(마크다운 본문)** 를 서버 컴포넌트로 추가. 직전 설교(#11) 패턴을 미러링하되 더 단순한 구조(필터 축소·행 레이아웃·미디어 없음)로 구현.

부수 작업으로, 설교·공지가 공유하는 **검색 입력 컴포넌트(`SearchPill`)** 를 추출해 중복을 제거.

**참조 이슈**: T11(공지 목록·상세) — 가이드 10장, OpenAPI `/api/notices`

---

### 🎯 구현 목표

- 목록 `/notices`: `GET /api/notices` — 고정글 우선 정렬, `q`(제목)·`tagId` 필터, 페이지네이션, 태그 필터
- 상세 `/notices/{id}`: `GET /api/notices/{id}` — 마크다운 본문, 조회수 매번 +1(`no-store`), 작성자 표기
- 콘텐츠 패칭 경계: 공개 페이지이므로 서버 컴포넌트 `fetch` 사용(목록 ISR, 상세 no-store)

---

### ✅ 구현 내용

#### 공지 API 클라이언트
- **파일**: `src/lib/api/notices.ts` (신규), `src/lib/api/types.ts` (수정)
- **변경 내용**: `NoticeListParams`·`buildNoticeQuery`·`getNotices`(목록)·`getNotice`(상세) 추가. `NoticeDetailResponse` 타입을 OpenAPI 스키마대로 정의
- **이유**: 설교 `sermons.ts`와 동일 구조로, 필터는 `q`·`tagId`·`page`만(설교의 preacher/series/from/to 제거)

#### 공유 검색 컴포넌트 추출
- **파일**: `src/components/common/SearchPill.tsx` (신규), `src/components/sermons/SermonSearch.tsx` (수정)
- **변경 내용**: 설교 검색의 범용 로직을 `SearchPill`로 추출(placeholder·aria-label 주입). `SermonSearch`는 문구만 넘겨 위임
- **이유**: 공지 검색과 설교 검색이 placeholder/라벨만 다르고 로직은 동일. 중복 대신 단일 컴포넌트로 통합

#### 공지 목록 페이지
- **파일**: `src/app/(site)/notices/page.tsx` (신규)
- **변경 내용**: `searchParams`(q·tagId·page) 파싱 → 목록·태그 병렬 fetch → `NoticeRow` 행 목록·`TagFilter`·`Pagination` 렌더. 고정글은 고정 배지 표시
- **이유**: 공지는 그리드 카드가 아닌 행(제목+날짜+고정 배지) 레이아웃(DESIGN.md `notice-row`)

#### 공지 상세 페이지
- **파일**: `src/app/(site)/notices/[id]/page.tsx` (신규)
- **변경 내용**: id 검증 → `getNotice` → 고정 배지·메타(날짜·조회수·작성자)·태그 클릭 필터·`MarkdownContent` 본문 렌더
- **이유**: 상세는 영상/오디오 없음. 조회수 +1 부수효과 때문에 `no-store`로 매번 호출

### 🔧 주요 변경사항 상세

#### 정렬은 서버를 신뢰
목록은 백엔드 기본 정렬(`isPinned,createdAt` 모두 desc → 고정글 우선·최신순)을 그대로 사용. `sort` 파라미터를 보내지 않고, 프론트에서 재정렬하지 않음.

**특이사항**: 고정글이 항상 상단에 오는 것은 서버 정렬 결과를 그대로 렌더해 보장. 테스트에서 고정글(id 5)이 일반글(id 4)보다 먼저 렌더되는지 순서를 직접 검증.

#### 캐시 전략
- 목록 `getNotices`: `next: { revalidate: 60 }` — 캐시 가능
- 상세 `getNotice`: `cache: 'no-store'` — 조회수 매번 +1(부수효과)이므로 캐시 금지

**특이사항**: 두 라우트 모두 동적 렌더(`ƒ`)로 분류됨(목록=searchParams 접근, 상세=no-store). 백엔드 없이도 빌드 시 fetch를 실행하지 않아 빌드 통과. 설교 라우트와 동일.

#### 검색은 제목만
검색창 placeholder를 "제목"으로 명시. `q`는 백엔드에서 제목만 매칭(가이드 10장 — Swagger 설명은 "제목/내용"이나 코드는 제목만이 사실). 내용 검색을 기대하게 만드는 UI를 두지 않음.

#### ActiveFilters 미사용
설교에는 숨은 필터(설교자·시리즈·기간)가 많아 활성 필터 칩 줄을 뒀으나, 공지는 `q`(검색창에 그대로 노출)·`tagId`(TagFilter에 선택 상태 노출)뿐이라 별도 칩 줄을 두지 않음(중복 제거).

#### SermonSearch 위임 (회귀 방지)
`SearchPill` 추출 후 `SermonSearch`는 위임만 하도록 교체. 기존 `SermonSearch.test.tsx`는 **수정 없이** green을 유지해 설교 검색 동작이 보존됐음을 증명.

### 🧪 테스트 및 검증

신규 테스트 4종(API·SearchPill·목록·상세) 추가, 각 도메인 검수 기준을 테스트로 게이트:

| 검수 기준 | 검증 테스트 |
|---|---|
| 고정글 항상 상단(재정렬 안 함) | 목록 페이지 테스트 — 서버 순서대로 렌더 |
| 상세 매번 백엔드 호출 | API 테스트 — `cache:'no-store'` 호출 단언 |
| 검색이 제목만 | SearchPill placeholder "제목" + 목록이 `q`만 전달 |

- 전체 스위트: **74개 파일 / 283개 테스트 통과**(설교 검색 회귀 포함)
- 린트: 이슈 없음
- 프로덕션 빌드: 성공(`/notices`·`/notices/[id]` 동적 라우트로 정상 분류)
