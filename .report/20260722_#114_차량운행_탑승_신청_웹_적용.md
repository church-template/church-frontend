# #114 차량운행 탑승 신청 웹 적용

### 📌 작업 개요

백엔드에 신설된 차량운행 도메인을 프론트에 적용. 회원(승인 교인)이 다가오는 운행일에 탑승을 신청·취소하고, 운영자가 운행일을 등록·수정·삭제하며 탑승 명단(연락처 포함)을 조회한다. 이어서 현재 위치 원탭 첨부, 카드 가독성 개선, 마이페이지 내 탑승 조회, 신청됨 상태 표기까지 확장했다.

전 화면 회원·어드민 영역(공개 노출 없음)이며, 지도 SDK·외부 키 없이 구현했다. 소스 36개 파일 변경(신규 16개), 순수 프론트 작업.

### 🎯 구현 목표

- 회원 탑승 신청·취소 (`/vehicle-runs`, 권한 `VEHICLE_APPLY`)
- 어드민 운행일 관리 + 운행일별 탑승 명단 (`/mypage/manage/vehicle-runs`, 권한 `VEHICLE_MANAGE`)
- 현재 위치(GPS) 원탭 첨부 → 카카오맵 핀 링크
- 차량 탑승 카드 가독성 개선 + 마이페이지 "내 차량 탑승" 섹션

### 🔧 주요 변경사항 상세

#### 백엔드 검증 게터 전송 차단
`docs/api-docs.json`의 `VehicleRequestCreateRequest` 스키마에 `pickupOrCoordinatesPresent`·`coordinatesPaired`·`coordinatesInRange`(boolean)가 노출되지만, 이는 Spring `@AssertTrue` 검증 게터가 OpenAPI에 샌 것이라 실제 입력 필드가 아니다.

**특이사항**:
- 요청 타입에 넣지 않고 `latitude`·`longitude`만 전송. 테스트가 `toHaveBeenCalledWith` 깊은 동등성으로 body에 두 키만 있음을 검증해 회귀를 차단.

#### 좌표 검증 미러(프론트) + 백엔드 최종 방어
백엔드가 "픽업 텍스트·좌표 중 최소 하나"를 요구하므로 신청 폼 zod에 동일 refine 하나만 둔다.

**특이사항**:
- 좌표 동반(둘 다/둘 다 없음)은 UI 불변식(첨부·지우기가 쌍으로 처리)으로 보장하고 백엔드가 최종 방어. 도달 불가능한 refine이 조용히 제출을 막는 함정을 피했다.
- geolocation 실패(권한 거부·타임아웃·미지원)는 문구 매핑 후 Toast로 안내하고 폼은 텍스트 입력으로 계속 사용 가능.

#### 마이페이지 섹션 데이터 소스
백엔드에 "내 차량 신청 이력" 전용 엔드포인트가 없어, 회원 목록 훅을 재사용해 다가오는 운행 중 내 신청(`myRequest≠null`)만 필터한다(캐시 키 공유로 중복 fetch 없음).

**특이사항**:
- 다가오는 운행 첫 페이지만 반영(지난 이력·다음 페이지 신청은 미표시) — DESIGN.md에 한계 명시.
- `useVehicleRuns`에 `enabled` 파라미터를 더해 권한 미보유 시 호출 억제(기본값 true라 기존 페이지 무영향).

#### 공유 상태 컴포넌트 추출
신청됨 상태를 카드·마이페이지에서 반복 수정하게 되어 `AppliedStatus`로 분리(체크 + primary 텍스트 인라인, `body-md` 600).

### 📌 참고사항
- 백엔드 계약 단일 진실은 `docs/api-docs.json`(차량운행 태그). 좌표 필드(`latitude`·`longitude`, nullable)와 `pickupLocation` 선택화가 반영됨.
- 회원 목록은 "다가오는 운행"만 반환 → 출발 시각이 지난 운행일은 회원 화면에 뜨지 않음(어드민은 지난 운행 포함 전체).
- 지난 탑승 이력 조회는 백엔드 전용 API 신설이 선행되어야 함(현재 범위 밖).
