# 어드민 03 일정 등록·수정·삭제

### 📌 작업 개요

어드민 트랙(2차 배치) wave2 도메인. 권한 보유 운영자가 **공개 캘린더·상세/모달에서 팝업 Dialog 폼**으로 일정을 등록·수정·삭제하도록 연동했다. 02의 `MarkdownEditor`·`TagMultiSelect`·인라인 액션 패턴·`.admin.ts` 분리 규칙을 재사용하고, `DateTimePicker`를 단독 생산했다.

### 🎯 구현 목표

- 공개 일정 캘린더·상세에 권한 보유 시 등록·수정·삭제 인라인 액션 노출
- 본문이 짧은 도메인이라 별도 화면 대신 팝업 Dialog 폼으로 입력
- 시작·종료 시간, 종일 토글, 장소, 본문, 태그 입력
- 종료가 시작보다 이후인지 검증
- 동시 수정 충돌(낙관락) 안내, KST 일관 표시
- 캘린더 클릭 모달에서도 수정·삭제 진입

### ✅ 구현 내용

#### DateTimePicker (03 단독 생산)
- **파일**: `src/components/admin/DateTimePicker.tsx`
- **변경 내용**: 네이티브 `<input type="datetime-local">`(종일이면 `type="date"`) 래퍼. `Input` 토큰·포커스링·error 배선 상속. 종일 시 표시값을 날짜로 슬라이스
- **이유**: `date-fns` 미설치·미사용(캘린더 코어도 `Date.UTC`+`Intl`로 구현) → 라이브러리 추가 없이 네이티브 입력으로 처리

#### 날짜 직렬화 유틸
- **파일**: `src/lib/date.ts`(`toServerDateTime`·`toLocalInput` 추가)
- **변경 내용**: 폼의 `datetime-local`/`date` 값 ↔ 서버 offset 없는 LocalDateTime 문자열 변환. 쓰기는 offset을 붙이지 않고 KST naive 그대로(읽기 `parseServerDate`의 +09:00 부착과 대칭)
- **이유**: 코드베이스에 쓰기 방향 직렬화 함수가 없어 신규 작성. `datetime-local`은 분 단위라 `:00` 보강, 종일은 `T00:00:00`

#### 일정 어드민 CRUD API
- **파일**: `src/lib/api/events.admin.ts` (신규)
- **변경 내용**: `EventCreateRequest`(필수 title·startAt)·`EventUpdateRequest`(+version) 타입과 `createEvent`(POST)/`updateEvent`(PUT)/`deleteEvent`(DELETE). `apiMutate` 래핑
- **이유**: 공개 GET 모듈(`events.ts`)을 RSC가 import하므로, 어드민 쓰기는 `.admin.ts`로 분리(`authStore` 클라이언트 체인이 서버 번들에 끌려가는 것 차단). `endAfterStart`·`patchEvent`는 불필요해 제외(YAGNI)

#### 일정 폼 + 팝업 Dialog

### 🔧 주요 변경사항 상세

#### KST naive 직렬화 (offset 미부착 대칭)
읽기 경로 `parseServerDate`는 서버 LocalDateTime에 `+09:00`을 붙여 KST로 파싱한다. 쓰기 경로는 반대로 **offset을 붙이지 않는다** — `datetime-local`의 naive 값과 서버 LocalDateTime이 둘 다 "KST 벽시계"로 동일하므로, `Date` 변환을 거치면 오히려 런타임 타임존이 섞여 어긋난다. `toServerDateTime` 후 `parseServerDate`가 동일 KST로 읽는지 round-trip 테스트로 검증했다(10:00 KST = 01:00 UTC).

#### 상세 모달 액션화 (타입 안전 주입)
`EventDetailModal`은 칩 클릭 시 단건 fetch한 `EventDetailResponse`(version 포함)를 보유한다. `shown ? <View/> : <Skeleton/>` 삼항의 truthy 가지 안에 `EventDetailActions`를 함께 넣어 `shown`이 non-null로 좁혀진 상태로 전달했다.

#### 기존 테스트 회귀 방지
캘린더 페이지·상세·모달에 새 client island(useMe·QueryClient 의존)를 주입하면서 기존 테스트 3개가 깨질 수 있어, 각 테스트에 `EventAdminActions` null-스텁(`() => null`)을 추가했다(02 선례).

**특이사항**:
- `EventFormDialog`의 RHF `watch("allDay")`가 React Compiler `react-hooks/incompatible-library` **경고**(error 아님)를 낸다. 종일 토글 실시간 반영을 위한 RHF 내재 특성으로, 빌드·타입·테스트에 영향 없음
- `endAfterStart`(요청 스키마의 bool)는 서버 파생 검증값으로 추정되어 요청에서 제외. 종료>시작은 클라 zod로 1차 검증(서버도 검증)
