# T05 인증 인프라 설계 문서 리뷰

대상: `docs/superpowers/specs/2026-06-11-t05-auth-infra-design.md`

## 결론

전체 방향은 `docs/church-frontend-guide.md`와 `docs/api-docs.json`의 인증 모델과 대체로 맞다. 다만 구현 계약이 모호한 지점이 있어 그대로 구현하면 인증 요청 경로, refresh 실패 처리, React Query 에러 처리, 로그아웃 정리에서 버그가 날 가능성이 높다.

전체 평가: Request Changes

이슈 통계: Critical 0개 / Major 5개 / Minor 2개

## Major

### 1. `authFetch` URL 결합 책임이 충돌한다

`docs/superpowers/specs/2026-06-11-t05-auth-infra-design.md:89`는 `authFetch`가 `apiUrl()`로 `NEXT_PUBLIC_API_BASE`를 결합한다고 설명한다. 그런데 `docs/superpowers/specs/2026-06-11-t05-auth-infra-design.md:106`은 호출자가 이미 `apiUrl('/api/members/me')`를 넘긴다. `docs/superpowers/specs/2026-06-11-t05-auth-infra-design.md:95`의 `login`도 raw fetch에 `/api/auth/login`만 적혀 있어 공개 API 함수의 base 결합 책임도 불명확하다.

영향: 구현자가 `authFetch` 내부와 호출부 양쪽에서 base를 결합하면 `http://localhost:8080/http://localhost:8080/...` 형태가 될 수 있고, 반대로 어느 쪽도 결합하지 않으면 Next origin `:3000`으로 잘못 호출된다.

제안: 계약을 하나로 고정한다.

- `authFetch(path: string, init?: RequestInit)`는 항상 `/api/...` path만 받고 내부에서 `apiUrl(path)`를 적용한다.
- raw `login/signup/refresh`도 `fetch(apiUrl('/api/auth/...'))`를 명시한다.
- `useMe` 예시는 `authFetch('/api/members/me')`로 수정한다.

### 2. refresh 실패 시 `authFetch`의 반환/throw 계약이 모순된다

`docs/superpowers/specs/2026-06-11-t05-auth-infra-design.md:88`은 refresh 실패 시 `forceLogout()` 후 원 401 응답을 반환한다고 한다. 하지만 `docs/church-frontend-guide.md:100`부터 `docs/church-frontend-guide.md:110`의 기준 스니펫은 공유 promise 내부에서 실패 시 throw하는 구조이고, 설계 문서의 `refreshing: Promise<string>` 타입(`docs/superpowers/specs/2026-06-11-t05-auth-infra-design.md:78`)도 실패 시 await 지점에서 reject되는 흐름이 자연스럽다.

영향: 동시 401 요청들이 refresh 실패 시 일부는 throw, 일부는 원 응답 반환처럼 섞일 수 있다. React Query 쪽도 실패를 `Error`로 받을지 `Response`로 받을지 예측하기 어렵다.

제안: 둘 중 하나를 문서에 고정한다. 권장안은 refresh promise는 reject를 허용하되, 각 원요청 쪽에서 catch하여 `forceLogout()` 후 원 401 `Response`를 반환하는 방식이다. 테스트에도 "refresh 실패 시 호출자에게 원 401이 반환된다" 또는 "ApiError가 throw된다" 중 하나를 명시한다.

### 3. `useMe`가 비-2xx 응답을 성공 데이터로 캐시할 수 있다

`docs/superpowers/specs/2026-06-11-t05-auth-infra-design.md:106`은 `authFetch(...).then(...)` 형태만 제시하고, `docs/superpowers/specs/2026-06-11-t05-auth-infra-design.md:109`는 401/403 errorCode를 surface한다고만 적는다. 그러나 `fetch`는 401/403에서 throw하지 않는다.

영향: `then(res => res.json())`만 구현하면 403 `ErrorResponse`가 `MeResponse`처럼 query `data`에 들어갈 수 있다. 그러면 `hasPermission`은 false를 반환하더라도 화면은 "권한 없음"과 "아직 로딩/데이터 없음"을 구분하기 어렵고, 401/403 분기 UX도 깨진다.

제안: `authFetch` 또는 별도 helper가 `!res.ok`일 때 `ApiError { status, errorCode, detail }`를 throw하도록 명시한다. `useMe`는 성공 시에만 `MeResponse`를 반환하고, 401/403은 query `error`로 분기한다.

### 4. `signOut()`이 네트워크/토큰 실패 시 local state를 못 비울 수 있다

`docs/superpowers/specs/2026-06-11-t05-auth-infra-design.md:97`은 `signOut()`이 logout 호출 후 `clear()`한다고 설명한다. 하지만 로그아웃 버튼은 사용자의 로컬 세션을 지우는 동작이므로, 서버 요청이 401/네트워크 오류/refresh 실패가 되어도 클라이언트 토큰은 반드시 제거되어야 한다.

영향: refresh token이 만료된 상태에서 사용자가 로그아웃을 누르면 서버 호출 실패 때문에 `clear()`가 실행되지 않고, 앱이 계속 로그인 상태처럼 보일 수 있다.

제안: `signOut()`은 서버 revoke 시도를 `try`로 감싸고 `finally { clear(); queryClient?.clear/removeQueries? }` 또는 최소 `clear()`를 보장한다고 문서화한다. 서버 204 멱등성과 별개로 클라이언트 정리는 항상 수행해야 한다.

### 5. Vitest 도입 범위가 Next 16 문서 기준으로 부족하다

`docs/superpowers/specs/2026-06-11-t05-auth-infra-design.md:138`은 `vitest` 도입만 언급한다. 현재 `package.json:5`부터 `package.json:10`에는 test script가 없고, `package.json:25`부터 `package.json:33`에도 vitest 관련 devDependency가 없다. Next 16 번들 문서 `node_modules/next/dist/docs/01-app/02-guides/testing/vitest.md:33`부터 `node_modules/next/dist/docs/01-app/02-guides/testing/vitest.md:45`는 TypeScript 수동 설정에 `vitest`, `@vitejs/plugin-react`, `jsdom`, `@testing-library/react`, `@testing-library/dom`, `vite-tsconfig-paths`를 제시하고, `node_modules/next/dist/docs/01-app/02-guides/testing/vitest.md:63`부터 config 파일을 요구한다.

영향: T05 완료 조건의 "검수 4종 vitest 통과"가 재현 가능한 실행 명령으로 연결되지 않는다. 특히 Zustand persist/localStorage를 테스트하면 jsdom 또는 명시적 mock이 필요하다.

제안: 완료 조건에 `vitest.config.mts`, `test` script, 필요한 devDependency 범위를 추가한다. 순수 `authFetch`만 node 환경에서 테스트할 계획이면 localStorage/store mock 전략을 별도로 명시한다.

## Minor

### 1. Provider 변경 범위가 기존 `Toaster`와 충돌할 수 있다

`docs/superpowers/specs/2026-06-11-t05-auth-infra-design.md:48`부터 `docs/superpowers/specs/2026-06-11-t05-auth-infra-design.md:50`은 `providers.tsx`를 만들고 `layout.tsx`에서 children을 감싼다고만 한다. 현재 `src/app/layout.tsx:27`부터 `src/app/layout.tsx:30`에는 `{children}`과 `<Toaster />`가 직접 배치되어 있다. Next 16 문서도 Provider는 client component로 만들고 가능한 깊게 `{children}`만 감싸라고 한다(`node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md:347`부터 `node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md:417`).

제안: `layout.tsx` 변경 예시를 "`<Providers>{children}</Providers>`를 추가하되 `<Toaster />`는 유지"로 명시한다.

### 2. refresh 대상 조건에 refreshToken 부재 케이스가 없다

`docs/superpowers/specs/2026-06-11-t05-auth-infra-design.md:85`는 `INVALID_TOKEN`이면 refresh 대상이라고만 한다. 비로그인 사용자가 보호 API를 실수로 호출하거나 persist hydration 전 토큰이 없으면 refreshToken 없이 refresh 요청을 보낼 수 있다.

제안: `INVALID_TOKEN`이어도 `refreshToken`이 없으면 refresh를 시도하지 않고 `forceLogout()` 후 원 응답을 반환/throw한다고 명시한다. 테스트에 "refreshToken 없음 → refresh 미호출" 케이스를 추가하면 좋다.

## Positive

- `docs/api-docs.json` 기준 Login/Refresh/Logout/Me DTO 이해는 정확하다.
- `AUTHENTICATION_FAILED`와 `INVALID_TOKEN`을 `status`가 아니라 `errorCode`로 분기하는 방향은 가이드와 일치한다.
- 권한 판단을 `roles`나 직분이 아니라 `MeResponse.permissions`로 제한한 점은 맞다.
- refresh 큐잉을 `authFetch`에만 두고 중복 구현을 금지한 점은 구현 복잡도를 줄인다.

## 핵심 수정 제안

1. `authFetch` 입력 계약을 path-only로 고정하고 모든 예시를 맞춘다.
2. `authFetch`/`useMe`의 HTTP 에러 처리 계약을 `ApiError throw` 또는 `Response 반환` 중 하나로 통일한다.
3. `signOut()`은 서버 요청 결과와 무관하게 local store를 반드시 clear하도록 완료 조건에 추가한다.
4. Vitest 실행 가능 상태까지 의존성/config/script를 완료 조건에 넣는다.
