# T03 시각 컴포넌트 설계 문서 리뷰

대상: `docs/superpowers/specs/2026-06-10-visual-components-design.md`

## Major

### 1. 토큰만 사용 조건과 픽셀 일치 조건이 충돌함

`docs/superpowers/specs/2026-06-10-visual-components-design.md:12-13`, `:51-54`, `:90-97`, `:132-137`

문서는 모든 값을 T02 토큰/Tailwind 유틸로 구현하고 hex/px 인라인을 0으로 요구하지만, 현재 `src/app/globals.css`에는 `11px`, `19px`, `14px`, `44px`, `56px`를 직접 표현할 컴포넌트 치수 토큰이 없다. 특히 `button-outline-on-dark`의 `11px 19px`, `text-input`의 `14px 16px`, 버튼 높이 `44/56px`는 기본 semantic spacing 토큰만으로는 픽셀 단위 일치가 어렵다.

영향: 구현자가 `py-[11px]`, `px-[19px]`, `h-[44px]` 같은 arbitrary px를 쓰면 문서의 "px 인라인 0" 게이트를 위반하고, 반대로 기존 spacing 토큰에 맞추면 DESIGN.md 픽셀 일치 조건을 위반한다.

제안: `globals.css @theme`에 컴포넌트 치수 토큰을 추가하도록 설계를 확장한다. 예: `--spacing-button-h`, `--spacing-button-cta-h`, `--spacing-button-outline-y`, `--spacing-button-outline-x`, `--spacing-input-y`. 또는 Tailwind 기본 spacing scale 사용 허용 여부를 명시하고, 허용한다면 "T02 토큰만" 문구를 완화한다.

### 2. 링크형 카드의 focus-visible 명세가 없음

`docs/superpowers/specs/2026-06-10-visual-components-design.md:110`, `:114-123`, `:150-156`

문서는 `SermonCard`, `NoticeRow`, `EventCard`에 `href`/행 전체 클릭을 두면서 hover 효과만 정의한다. 완료 게이트는 모든 인터랙티브 요소의 focus-visible 링과 키보드 조작을 요구하지만, 카드 루트 또는 내부 `Link`에 어떤 focus class를 적용할지 명시하지 않았다.

영향: hover는 통과해도 키보드 Tab 검수에서 포커스 위치가 안 보이거나, 카드 전체 클릭 영역과 포커스 영역이 어긋날 수 있다.

제안: `href`가 있는 합성 카드는 `Link`를 루트로 렌더하고 `focus-visible:ring-2 focus-visible:ring-primary focus-visible:ring-offset-2 focus-visible:ring-offset-canvas`를 명시한다. `href`가 없으면 비인터랙티브 `article/div`로 렌더하고 hover/focus 스타일을 끈다는 조건도 함께 적는다.

### 3. `/showcase` 프로덕션 차단 위치가 위험함

`docs/superpowers/specs/2026-06-10-visual-components-design.md:145-148`

문서는 "파일 상단에서 `if (process.env.NODE_ENV === "production") notFound();`"를 지시한다. Next 문서상 `notFound()`는 호출 시 라우트 세그먼트 렌더링을 종료하는 예외를 던지는 함수다. 모듈 top-level에서 던지면 페이지 렌더 함수가 아니라 모듈 평가 시점에 예외가 발생할 수 있어 production build/import 단계에서 실패할 위험이 있다.

제안: import 아래 top-level이 아니라 `Page` 컴포넌트 본문 첫 줄에서 호출하도록 바꾼다.

```tsx
export default function ShowcasePage() {
  if (process.env.NODE_ENV === "production") notFound();
  return (...);
}
```

### 4. Button disabled 상태가 variant별로 불명확함

`docs/superpowers/specs/2026-06-10-visual-components-design.md:98`

문서가 `disabled:bg-primary-disabled`를 공통 상태처럼 적고 있다. 이를 base class로 넣으면 secondary, outlineOnDark, tertiary까지 disabled 시 primary-disabled 배경을 갖게 되어 각 variant의 배경 규칙과 충돌한다.

제안: disabled 스타일을 variant별로 분리한다. 최소한 primary/pillCta만 `bg-primary-disabled text-on-primary`, secondary는 `bg-surface-strong text-muted`, outlineOnDark/tertiary는 배경을 투명하게 유지하는 식으로 명시한다.

## Minor

### 1. Input 오류 메시지 접근성 연결이 빠져 있음

`docs/superpowers/specs/2026-06-10-visual-components-design.md:132-139`

`aria-invalid="true"`만 있고 오류 메시지와 input의 연결(`aria-describedby`)이 없다. `error`가 있을 때 message id를 생성하거나 `id` 기반으로 `${id}-error`를 연결하도록 명세하면 스크린리더 검수까지 안정적이다.

### 2. `thumbnailUrl`의 허용 URL 범위가 불명확함

`docs/superpowers/specs/2026-06-10-visual-components-design.md:114`, `:148`

`remotePatterns`를 T10 범위 밖으로 둔 것은 타당하지만, `SermonCard` prop이 `thumbnailUrl`이라 외부 URL도 가능한 것처럼 보인다. Next `Image`는 외부 URL에 remotePatterns가 필요하므로 T03에서는 local path 또는 same-origin `/api/media/{id}`만 지원한다고 명시하는 편이 안전하다.

## Positive

- shadcn 도입 범위를 동작 컴포넌트로 제한하고 Button/Card/Badge/Input을 직접 구현한다는 결정은 가이드 15.1과 일치한다.
- `cn()` 위치를 `src/lib/utils.ts`로 잡은 것은 이후 shadcn 컴포넌트 도입 경로와 호환된다.
- Card를 variant 하나로 비대하게 만들지 않고 베이스 + 합성 컴포넌트로 나눈 방향은 현재 T03 범위에 적절하다.
