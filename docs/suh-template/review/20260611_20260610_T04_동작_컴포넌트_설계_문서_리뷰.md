# T04 동작 컴포넌트 설계 문서 리뷰

대상: `docs/superpowers/specs/2026-06-11-behavior-components-design.md`

## Major

### 1. `components/ui/` 목록 점검 완료 조건이 현재 저장소와 충돌함

`docs/superpowers/specs/2026-06-11-behavior-components-design.md:134`, `:167`

문서는 목록 점검을 "`components/ui/`에 15.1 목록 외 shadcn 컴포넌트 없음"으로 적었다가, 완료 조건에서는 "`components/ui/`에 15.1 목록 외 컴포넌트 없음"으로 좁혀 적고 있다. 현재 저장소에는 T03 산출물인 `Button.tsx`, `Card.tsx`, `Badge.tsx`, `Input.tsx`가 이미 `src/components/ui/`에 존재하므로, 완료 조건을 문자 그대로 적용하면 T04는 통과할 수 없다.

영향: 구현자가 T03 직접 구현 컴포넌트를 삭제하거나, 반대로 검수자가 완료 조건을 잘못 해석해 불필요한 실패를 만들 수 있다.

제안: 완료 조건도 7.2와 동일하게 "15.1 목록 외 **shadcn 도입 컴포넌트** 없음"으로 수정한다. 검증은 파일명 전체 금지가 아니라, lowercase shadcn 도입분만 allowlist(`dialog.tsx`, `sheet.tsx`, `popover.tsx`, `select.tsx`, `dropdown-menu.tsx`, `tabs.tsx`, `sonner.tsx`)로 점검하는 식이 안전하다.

### 2. CLI 우선 전략과 `components.json` 최소 생성 방침이 불안정함

`docs/superpowers/specs/2026-06-11-behavior-components-design.md:43-47`

공식 shadcn 문서는 기존 프로젝트에서 먼저 `init`을 실행하고, CLI로 컴포넌트를 추가할 때 `components.json`을 프로젝트 설정 이해와 생성 커스터마이즈에 사용한다고 설명한다. 또한 `components.json`의 `tailwind.css`, `tailwind.cssVariables`, `rsc`, `tsx`, `aliases` 등은 실제 생성 결과에 영향을 준다. 그런데 설계는 "alias만" 있는 최소 `components.json`을 만들겠다고 되어 있어 CLI가 실패하거나, 기본 테마/변수/스타일을 의도치 않게 생성할 여지가 남는다.

영향: T04의 핵심 금지 조건인 shadcn 시맨틱 변수 레이어 미도입, globals.css 자동 수정 금지, zinc 잔재 제거가 CLI 실행 단계에서 깨질 수 있다.

제안: 둘 중 하나로 명확히 고정한다.

1. CLI를 쓰려면 `components.json` 전체 값을 명시한다: `style: "new-york"`, `rsc: true`, `tsx: true`, `tailwind.css: "src/app/globals.css"`, `tailwind.config: ""`, `tailwind.cssVariables: false`, `aliases` 전체.
2. 더 보수적으로는 CLI를 의존성/레지스트리 확인용으로만 쓰고, 컴포넌트 파일은 레지스트리 소스 수동 복사로 고정한다.

### 3. Playwright 검증 범위가 현재 의존성/스크립트에 반영되지 않음

`docs/superpowers/specs/2026-06-11-behavior-components-design.md:130-137`, `package.json`

설계는 Playwright 자동 점검을 완료 게이트로 요구하지만, 의존성 목록에는 `@playwright/test`가 없고 `package.json`에도 `test` 또는 `test:e2e` 스크립트가 없다. 현재 스크립트는 `dev`, `build`, `start`, `lint`뿐이다.

영향: 구현 완료 후 "Playwright 자동 점검"을 실제로 실행할 방법이 없어 완료 조건이 문서상으로만 남는다.

제안: T04 포함 범위에 Playwright 도입을 넣을지 결정해야 한다. 포함한다면 devDependency와 스크립트, 최소 config, `/showcase` 기반 테스트 파일 경로를 명세한다. 포함하지 않는다면 7.2를 "수동 브라우저 점검 + 추후 테스트 태스크"로 낮춰야 한다.

### 4. Toast의 스크린리더 낭독을 자동 테스트로 단정하고 있음

`docs/superpowers/specs/2026-06-11-behavior-components-design.md:132`, `:166`

"새 토스트가 `aria-live` 노드에 들어가 SR이 낭독한다"는 표현은 자동 테스트로 검증 가능한 범위를 넘는다. Playwright는 DOM에 live region이 있고 toast 텍스트가 들어가는지는 확인할 수 있지만, VoiceOver/NVDA가 실제로 낭독했는지는 일반 E2E에서 신뢰성 있게 단정하기 어렵다.

영향: 테스트가 거짓 확신을 주거나, 반대로 자동화가 불가능한 조건 때문에 완료 판단이 흔들린다.

제안: 자동 검증은 `role`/`aria-live`/DOM 삽입/4초 후 제거까지로 제한하고, 실제 낭독은 "VoiceOver 수동 스모크"로 분리한다.

### 5. Portal 계층의 z-index 정책이 빠져 있음

`docs/superpowers/specs/2026-06-11-behavior-components-design.md:73-87`, `:93-110`

Dialog, Sheet, Popover, DropdownMenu, Select는 모두 portal/floating layer를 만들지만 설계에는 overlay/content의 z-index 기준이 없다. 현재 DESIGN/토큰에는 z-index 토큰도 없고, 기존 네비는 가이드상 fixed z-index를 가진다. z-index가 명시되지 않으면 모달 오버레이나 메뉴가 네비 아래에 깔릴 수 있다.

영향: a11y 동작은 살아 있어도 실제 화면에서는 클릭/시각 계층이 깨질 수 있다.

제안: T04 범위에서 최소 계층 정책을 추가한다. 예: `z-50`을 동작 컴포넌트 portal 계층의 표준으로 허용하거나, `--z-overlay`/`--z-popover` 토큰을 먼저 정의한다. Dialog/Sheet overlay와 content, Select/Dropdown/Popover content에 동일한 기준을 적용한다.

## Minor

### 1. `tw-animate-css` 미도입 결정은 좋지만, 필요한 keyframe 수가 과소 산정됨

`docs/superpowers/specs/2026-06-11-behavior-components-design.md:35`, `:118`

Sheet의 `slide-in-{right,left,top,bottom}`만 적혀 있고 닫힘용 `slide-out-*`가 빠져 있다. 실제로 `data-state=closed` 애니메이션까지 맞추려면 방향별 in/out이 필요하다.

제안: keyframe 목록에 `slide-out-{right,left,top,bottom}`를 추가하거나, translate sign만 바꾸는 공통 keyframe/utility 조합으로 구현한다고 명시한다.

### 2. Dialog/Sheet의 accessible description 정책이 없음

`docs/superpowers/specs/2026-06-11-behavior-components-design.md:93-100`

`DialogTitle` 보존은 명확하지만 `DialogDescription`을 언제 유지하고 언제 제거할지 정책이 없다. Radix는 설명을 제거할 경우 `aria-describedby={undefined}` 처리가 필요할 수 있다고 안내한다.

제안: 기본은 `DialogDescription`을 제공하고, 설명이 없는 모달만 `aria-describedby={undefined}`를 명시하도록 컴포넌트 API 또는 사용 규칙에 추가한다.

### 3. `notify.ts`의 호출 경계가 불명확함

`docs/superpowers/specs/2026-06-11-behavior-components-design.md:105`

`notify.success/error`는 Sonner의 client-side toast API를 감싸는 seam인데, 서버 컴포넌트에서 import하거나 호출하지 않는다는 경계가 없다.

제안: `src/lib/notify.ts`는 client component/event handler 전용이라고 명시하고, 서버 액션/서버 컴포넌트에서는 직접 호출하지 않는다는 사용 규칙을 추가한다.

### 4. 정적 검증 명령이 package script와 맞지 않음

`docs/superpowers/specs/2026-06-11-behavior-components-design.md:137`

문서는 `tsc --noEmit`을 적었지만 `package.json`에는 `typecheck` 스크립트가 없다. 실행 자체는 `pnpm exec tsc --noEmit`로 가능하지만, 완료 게이트로 반복 실행할 명령은 스크립트화하는 편이 낫다.

제안: T04에서 `typecheck` 스크립트를 추가할지, 아니면 검증 명령을 `pnpm exec tsc --noEmit`로 명확히 쓸지 정한다.

## Positive

- 동작 컴포넌트만 shadcn/Radix로 가져오고 T03 시각 컴포넌트와 분리하는 방향은 가이드 15.1과 일치한다.
- shadcn 변수 레이어를 도입하지 않고 DESIGN 토큰으로 직접 재스킨하겠다는 결정은 이 프로젝트의 "토큰 어휘 단일화" 원칙에 맞다.
- DialogTitle 보존, 포커스 트랩/복귀, ESC/외부 클릭, body scroll lock 등 T04 이슈의 a11y 계약을 설계 단계에서 완료 조건으로 끌어올린 점은 좋다.
- `notify` seam으로 Toast 출력 채널을 단일화하는 방향은 T06의 errorCode 기반 UI 매핑과 잘 맞는다.
